# Q2: Three-Environment Architecture Evolution
 
## Overview
 
The current deployment is a single t3.xlarge EC2 instance in eu-west-1 running everything via Docker Compose with Caddy and sslip.io. This document proposes how the stack evolves across dev, stage, and prod, introducing AWS managed services progressively to keep costs and complexity proportional to what each environment actually needs.
 
---
 
## Per-Environment Component Table
 
| Component | Dev | Stage | Prod |
|---|---|---|---|
| **LobeChat** | Container on t3.medium (single EC2) | Container on t3.xlarge (same as current) | Container on t3.2xlarge; ECS Fargate evaluated and rejected (see note below) |
| **Casdoor** | Container, same EC2 | Container, same EC2 | Container on LobeChat EC2; Cognito evaluated but Casdoor kept for MCP webhook compatibility |
| **PostgreSQL** | Container (pgvector/pg16), local bind-mount | RDS PostgreSQL 16, db.t3.micro, single-AZ | RDS PostgreSQL 16, db.r5.large, Multi-AZ, 35-day automated backup |
| **MinIO** | Container, local bind-mount | Amazon S3 (native) | Amazon S3 with versioning + Glacier lifecycle |
| **Qdrant** | Container on t3.medium, 10 GB EBS | Container on t3.xlarge, separate 30 GB gp3 EBS | Container on dedicated r5.large, 100 GB gp3 EBS encrypted, DLM snapshots |
| **MCPHub** | Container, same EC2 | Container, same EC2 | Container on LobeChat EC2 |
| **Hayhooks** | Container, same EC2 | Container, same EC2 | Container on LobeChat EC2 |
| **vLLM** | Disabled (no GPU) | Disabled | Disabled; OpenRouter / AWS Bedrock used for inference |
| **Reverse proxy / TLS** | Caddy + sslip.io (Let's Encrypt) | ALB + ACM (wildcard cert, no rate-limit risk) | ALB + ACM + Route53 (custom domain) |
| **Secrets** | `.env` file locally | AWS Secrets Manager | AWS Secrets Manager with automatic rotation |
| **Observability** | Docker logs only | CloudWatch Logs agent | CloudWatch Logs + Metrics + Alarms + Dashboard |
| **Container registry** | Local images / Docker Hub | ECR | ECR (image scanning enabled) |
 
> **Why not ECS Fargate for LobeChat in prod?** Fargate doesn't support bind-mounting local directories, and this stack patches `route.js` via a volume mount. Moving to Fargate would require building a custom image. That's a valid future refactor, but it's out of scope here.

## Hard Constraint: Qdrant on EC2 in All Three Environments
 
Qdrant runs on EC2 in every environment. Managed alternatives were evaluated and rejected:
 
- **Pinecone serverless**: $0.033/1K reads x 440,000 queries/month at 1000 MAU = $14,520/month vs. ~$100/month for an r5.large. Over 100x more expensive at scale.
- **OpenSearch k-NN**: minimum 2-node cluster in eu-west-1 costs ~$168/month with less flexible index configuration than native Qdrant.
- **Weaviate Cloud**: not available natively on AWS eu-west-1; data residency risk for EU-regulated workloads.
The numbers aren't close. Self-hosted Qdrant on EC2 is the only option that makes sense financially and operationally at any scale this deployment is likely to see.
 
**EBS sizing per environment:**
 
| Env | EBS volume | Rationale |
|---|---|---|
| Dev | 10 GB gp3 (bind-mount on root volume) | Small test dataset, no production vectors |
| Stage | 30 GB gp3 (separate attached volume) | Anonymised prod snapshot, ~5M vectors max |
| Prod | 100 GB gp3 encrypted (separate attached volume) | 50M vectors at 1536 dimensions x 4 bytes = ~300 GB raw; HNSW index overhead ~20%; 100 GB covers ~6 months of growth |
 
**Snapshot policy (prod):** AWS Data Lifecycle Manager (DLM) rule runs daily EBS snapshots at 02:00 UTC, 7-day retention. Cost: $0.05/GB/month for snapshot storage = ~$5/month for 100 GB.
 
**Instance recovery (prod):** A CloudWatch alarm on `StatusCheckFailed_System` triggers an EC2 instance recovery action. This preserves the EBS volume and EIP attachment, so Qdrant data survives. Recovery SLO is under 5 minutes. If the volume itself corrupts, the fallback is restoring from the previous day's DLM snapshot and re-indexing from S3, accepting up to 24 hours of index staleness.

## AWS Managed Services in Prod (7 total)
 
1. **RDS PostgreSQL 16** replaces the containerised pgvector instance. Multi-AZ automatic failover under 60 seconds, automated backups with 35-day retention, point-in-time recovery to 5-minute granularity. No WAL management overhead.
2. **Amazon S3** replaces MinIO. Versioning preserves file history; a lifecycle rule moves objects older than 90 days to S3 Glacier Instant Retrieval ($0.004/GB/month vs. $0.023/GB for Standard), cutting long-term storage cost by 83%.
3. **ALB + ACM** replaces Caddy. ACM issues and auto-renews the TLS certificate with no Let's Encrypt rate-limit risk. ALB health checks enable zero-downtime deploys via weighted target groups.
4. **AWS Secrets Manager** replaces the `.env` file. Secrets are fetched at container startup via the AWS SDK; RDS credentials rotate automatically via a Lambda rotation function with a full audit trail in CloudTrail.
5. **Amazon Route53** provides DNS for a custom domain (e.g., `chat.iberia-legal.com`). Alias record points to the ALB; health-check failover routes to a static maintenance page if the ALB target goes unhealthy.
6. **Amazon CloudWatch** centralises log aggregation, custom metrics (active users, token spend, Qdrant query latency p99), and alarms for error rate above 1% and latency p99 above 3 seconds.
7. **Amazon ECR** is the private container registry. Images tagged with Git SHA and semantic version; image scanning on push detects CVEs before deployment.

## Promotion Flow: dev > stage > prod
 
```
feature/* branch
    |
    v  Pull Request review + automated tests (GitHub Actions)
develop branch
    |
    v  Auto-deploy on merge -> stage environment
    |  Integration tests: auth flow, MCP tool call, file upload, streaming
    v  Manual approval gate (lead engineer)
main branch  <-- squash merge from develop
    |
    v  cz bump -> version tag (e.g., v1.3.0)
    |  ECR image built + pushed (SHA + semver tag)
    v  GitHub Actions: terraform apply (prod) + rolling container restart
```
 
**Branching:** `feature/*` branches are short-lived, merged via PR into `develop`. `develop` is always deployable to stage. `main` only receives squash merges after manual approval. Production deploys are tag-triggered only — a push to `main` without a semver tag does nothing.
 
**Config promotion:** Environment-specific values (ALB ARNs, RDS endpoints, S3 bucket names) live in Terraform workspace state. Application config (`mcp_settings.json`, `Caddyfile`) is versioned in the repo and applied on each deploy via the user_data template or a post-deploy hook.

## Data Strategy
 
| Concern | Dev | Stage | Prod |
|---|---|---|---|
| Source data | Synthetic: generated contracts with fake names | Anonymised prod snapshot (weekly pg_dump + `pg_anonymizer`) | Live data |
| PII in Qdrant | No (synthetic) | No (anonymised before indexing) | Chunk text only; document UUID in payload; no raw PII |
| Backup | None | Weekly manual RDS snapshot | RDS: automated daily, 35-day retention, PITR. S3: versioning. Qdrant EBS: DLM daily, 7-day |
| Restore test | Not performed | Monthly restore drill to temporary RDS instance | Quarterly drill; documented RTO under 1 hour, RPO under 24 hours |

## Trade-off Table: Reliability vs. Cost vs. Ops Complexity
 
| Dimension | Dev | Stage | Prod |
|---|---|---|---|
| Availability target | Best-effort (under 99%) | ~99% (single-AZ RDS) | ~99.5% (Multi-AZ RDS + EC2 recovery) |
| Monthly infra cost | ~$60 | ~$350 | ~$700 |
| Ops complexity | Low: one EC2, `docker compose up` | Medium: Terraform manages RDS + ALB + Secrets | High: DLM, CloudWatch alarms, ECR lifecycle, Secrets rotation |
| Deployment risk | Low | Medium: caught here before prod | Managed: weighted ALB target groups for canary |
| Secret management | Weak (.env file) | Strong (Secrets Manager) | Strong + automatic rotation |
 
One trade-off worth calling out: introducing RDS at stage eliminates the containerised Postgres but adds a ~2-minute cold-start penalty when Terraform creates a new stage environment from scratch. That's fine because stage is long-lived. For PR-level preview environments, containerised Postgres on a t3.small is still the right call — a managed RDS per preview PR costs at least $49/month, which adds up fast.
